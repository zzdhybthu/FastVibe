# 有效 Prompt 模式: 与 Claude Code 协作的最佳实践

本文档总结了与 Claude Code (CC) 协作中验证有效的 prompt 模式。每个模式包含适用场景、模板和实际示例。

---

## 模式 1: 任务定义模式

**适用场景:** 提交任何开发任务时的基础框架。

一个完整的任务定义包含三个要素: 目标 (做什么)、约束 (怎么做的边界)、验证标准 (怎么算做好了)。

### 模板

```
## 目标
[用 1-2 句话描述要达成的结果]

## 约束
- [技术约束: 语言/框架/版本要求]
- [架构约束: 需要遵守的模式或规范]
- [兼容性约束: 不能破坏的现有功能]

## 验证标准
- [ ] [可检查的条件 1]
- [ ] [可检查的条件 2]
- [ ] [测试通过的标准]
```

### 示例

```
## 目标
为 /api/tasks 端点添加分页功能，支持 cursor-based pagination。

## 约束
- 使用现有的 FastAPI + SQLAlchemy 技术栈
- 遵循项目中 /api/users 端点的分页实现模式
- 不能修改现有 Task 模型的字段
- 默认每页 20 条，最大 100 条

## 验证标准
- [ ] GET /api/tasks?limit=10 返回最多 10 条记录
- [ ] 响应包含 next_cursor 字段用于获取下一页
- [ ] 当没有更多数据时 next_cursor 为 null
- [ ] GET /api/tasks?cursor=xxx&limit=10 正确返回下一页数据
- [ ] 现有的 POST/PUT/DELETE /api/tasks 端点不受影响
- [ ] pytest tests/test_tasks.py 全部通过
```

## 模式 2: 上下文注入模式

**适用场景:** AI 需要理解项目背景才能做好的任务。

单纯的 prompt 无法传递所有上下文。真正有效的上下文注入分为三层:

### 第一层: CLAUDE.md (持久上下文)

写在 CLAUDE.md 中的信息会在每次会话中自动加载，适合放:
- 项目架构和技术栈
- 代码规范和开发约定
- 已知陷阱和注意事项
- 工作流规则

### 第二层: PROGRESS.md (会话间记忆)

记录当前进度和待办事项，让 AI 在新会话中快速恢复上下文:

```markdown
# PROGRESS.md

## 当前状态
用户认证模块已完成，正在进行订单管理模块。

## 已完成
- [x] User model + CRUD
- [x] JWT 认证
- [x] 登录/注册 API

## 进行中
- [ ] Order model (已创建基本结构，缺少状态机)

## 待办
- [ ] 订单 API (CRUD + 状态流转)
- [ ] 支付集成

## 已知问题
- User.email 的索引在 SQLite 下不生效，部署到 PostgreSQL 后需要验证
```

### 第三层: 任务级上下文 (Prompt 中直接提供)

```
请参考以下文件来理解当前的代码模式:
- src/models/user.py -- 模型定义的范例
- src/api/users.py -- API 端点的范例
- tests/test_users.py -- 测试的范例

现在用同样的模式实现 Order 模块。

背景: 订单有状态流转 (created -> paid -> shipped -> delivered -> completed)，
状态只能正向流转，不能回退。
```

### 关键点

上下文注入的目的是**消除歧义**。当你发现 AI 的输出和你预期不同时，通常不是 AI 能力不够，而是上下文不足。与其说"你做错了"，不如补充缺失的上下文让 AI 重新理解。

## 模式 3: 分步指令模式

**适用场景:** 需要先规划再执行的复杂任务，或你想对中间结果有控制权的场景。

### 模板: Plan-Execute 模式

```
这个任务我们分两步走:

第一步 (规划): 请先分析需求并输出执行计划，包括:
1. 需要创建/修改哪些文件
2. 每个文件要做什么改动
3. 改动的先后顺序
4. 可能的风险点

不要开始写代码，我需要先审核计划。

需求: [需求描述]
```

审核计划后:

```
计划看起来不错，有一处调整:
- [你的修改意见]

请按照修改后的计划执行。
```

### 模板: Incremental 模式

用于特别大或风险高的变更:

```
我们需要把数据库从 SQLite 迁移到 PostgreSQL。
这个改动影响面大，请按以下步骤逐步进行，每一步完成后暂停等我确认:

步骤 1: 更新数据库连接配置，确保能连上 PostgreSQL
步骤 2: 修改 ORM 模型中 SQLite 特有的语法
步骤 3: 创建数据迁移脚本
步骤 4: 更新测试配置

先执行步骤 1。
```

### 何时使用分步 vs 一步到位

| 场景 | 方式 |
|---|---|
| 新功能开发 (独立模块) | 一步到位 |
| 重构 (影响多个文件) | 分步 |
| Bug 修复 (原因已知) | 一步到位 |
| Bug 修复 (原因未知) | 分步: 先定位再修复 |
| 数据库变更 | 分步 + Plan Mode |
| UI 组件开发 | 一步到位 |

## 模式 4: 错误修复模式

**适用场景:** 修复 bug 或处理错误。

好的 bug 报告 = 快速修复。给 AI 的 bug 描述应该和给人类开发者的一样完整。

### 模板

```
## Bug 描述
[简要描述什么功能出了问题]

## 复现步骤
1. [步骤 1]
2. [步骤 2]
3. [观察到的错误]

## 错误日志
```
[粘贴完整的错误日志/堆栈跟踪]
```

## 期望行为
[应该发生什么]

## 实际行为
[实际发生了什么]

## 已排查
- [你已经检查过的地方]
- [你的猜测 (如果有)]
```

### 示例

```
## Bug 描述
用户上传头像后，个人资料页面显示空白图片。

## 复现步骤
1. 登录账号 testuser
2. 进入 /profile/settings
3. 上传一张 JPEG 图片 (500KB)
4. 保存后刷新页面
5. 头像位置显示空白 (broken image icon)

## 错误日志
```
[api] 2024-03-15 10:23:45 ERROR: FileNotFoundError: /uploads/avatars/abc123.jpg
[api] Traceback:
  File "src/api/profile.py", line 87, in get_avatar
    return FileResponse(avatar_path)
```

## 期望行为
上传的图片应该正常显示在头像位置。

## 实际行为
图片上传 API 返回 200 成功，但文件没有保存到 /uploads/avatars/ 目录。

## 已排查
- 上传 API 确实返回了 200 和文件路径
- /uploads/avatars/ 目录存在且权限正确
- 可能是文件保存逻辑的问题，upload.py 中的 save_file() 函数
```

### 关键技巧

- **永远提供错误日志**: AI 解读堆栈跟踪的速度比你快
- **说明已排查的内容**: 避免 AI 重复检查你已经验证过的部分
- **提供复现环境**: "在 Docker 中运行" vs "本地 Mac 运行" 可能导致完全不同的根因

## 模式 5: 代码审查模式

**适用场景:** 让 AI 审查代码质量、找出潜在问题。

### 模板: 聚焦审查

```
请审查以下文件的改动，重点关注:

关注点:
- [安全性 / 性能 / 可维护性 / 正确性]

审查标准:
- [项目特定的标准，如 "所有数据库查询必须参数化"]
- [这个模块的特定要求，如 "这是支付模块，金额计算不能用浮点数"]

文件: [文件路径或 diff]

请按以下格式输出:
1. 严重问题 (必须修复)
2. 建议改进 (推荐修复)
3. 小问题 (可选修复)
```

### 示例

```
请审查 src/payment/processor.py 的改动，重点关注:

关注点:
- 安全性 (支付相关代码，不能有漏洞)
- 金额精度 (必须使用 Decimal，不能用 float)
- 错误处理 (支付失败必须有回滚机制)

审查标准:
- 所有金额运算使用 decimal.Decimal
- 支付状态变更必须在事务中完成
- 外部 API 调用必须有超时和重试
- 敏感信息 (卡号等) 不能出现在日志中

请按以下格式输出:
1. 严重问题 (必须修复)
2. 建议改进 (推荐修复)
3. 小问题 (可选修复)
```

### 模板: 全面审查

```
请全面审查 [文件/模块]，从以下维度评估:

1. 正确性: 逻辑是否正确，边界条件是否处理
2. 安全性: 是否有注入、越权、数据泄露风险
3. 性能: 是否有 N+1 查询、内存泄漏、不必要的计算
4. 可维护性: 命名是否清晰、函数是否过长、是否有重复代码
5. 测试: 测试覆盖是否充分、是否有遗漏的场景

对每个维度给出评分 (1-5) 和具体说明。
```

## 模式 6: 探索性任务模式

**适用场景:** 你不确定该怎么做、需要 AI 先调研的场景。

### 模板

```
我想在项目中实现 [功能/改进]，但不确定最佳方案。

当前情况:
- [项目的现状描述]
- [面临的具体挑战]

请分析以下几种可能的方案:
1. [方案 A 的大致思路]
2. [方案 B 的大致思路]
3. 或者你有更好的建议

对每个方案，请评估:
- 实现复杂度
- 对现有代码的影响
- 性能影响
- 长期维护成本

最后给出你的推荐和理由。不要开始实现，我需要先决定方案。
```

### 示例

```
我想给 VibeCoding 的 Web UI 添加实时日志流。当 CC 在执行任务时，
用户能在浏览器中实时看到输出。

当前情况:
- 后端是 FastAPI，前端是原生 JS (PWA)
- CC 的输出通过 subprocess 的 stdout 获取
- 目前是任务完成后一次性返回全部输出

请分析以下方案:
1. WebSocket: 后端维护 WS 连接，实时推送 CC 输出
2. SSE (Server-Sent Events): 后端使用 SSE 流式推送
3. 轮询: 前端每秒请求一次最新输出

对每个方案评估实现复杂度、性能、和对移动端的友好程度。
给出推荐方案。
```

## 模式组合的实际工作流

真实场景中，这些模式通常组合使用:

```
# 第一步: 探索 (模式 6)
"我需要实现 X 功能，不确定方案，请分析..."

# 第二步: 确定方案后，用 Plan Mode (模式 3)
"方案 B 不错。请基于方案 B 生成详细的执行计划..."

# 第三步: 审核计划后，提交执行任务 (模式 1)
"计划批准。请执行步骤 1-3。目标/约束/验证标准如下..."

# 第四步: 出问题了 (模式 4)
"步骤 2 的测试失败了，错误日志如下..."

# 第五步: 完成后审查 (模式 5)
"功能完成了，请做一次安全性审查..."
```

## 通用技巧

### 好 Prompt 的特征

1. **具体**: "添加输入验证" -> "对 email 字段添加格式校验，无效时返回 422"
2. **有上下文**: 引用现有代码、说明背景
3. **有边界**: 明确什么在范围内、什么不在
4. **可验证**: 有明确的"完成"标准

### 避免的表达

| 模糊表达 | 清晰表达 |
|---|---|
| "优化一下性能" | "这个查询在 1000 条数据时需要 3 秒，目标是降到 500ms 以内" |
| "改好看一点" | "按照 Tailwind UI 的 dashboard 模板风格调整布局" |
| "修复 bug" | "POST /api/orders 在 quantity=0 时返回 500 而不是 400" |
| "写个测试" | "为 OrderService.calculate_total() 写单元测试，覆盖折扣、免运费、边界值三种场景" |
| "重构代码" | "把 api.py 中超过 50 行的函数拆分成独立函数，保持 API 行为不变" |
