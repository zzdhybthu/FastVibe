# AI Servant Leadership: 管理 AI 编程代理的实践指南

## 什么是 AI Servant Leadership

传统软件开发管理中，管理者通过代码审查、任务分配、进度追踪来控制开发过程。当我们的"开发者"变成 AI 代理 (如 Claude Code) 时，管理方式需要根本性的转变。

**AI Servant Leadership** 是一种管理范式: 你不是 AI 的老板，你是 AI 的"服务者"。你的工作不是告诉 AI 每一步该怎么做，而是为 AI 创造成功完成任务的条件。

核心思想来自一个简单的认知:

> **Context, not control.** 提供上下文，而非控制过程。

AI 代理拥有巨大的代码能力，但它缺乏你头脑中的业务上下文、架构决策的历史原因、以及项目的隐性约定。你的价值在于填补这个信息差。

## 核心原则

### 原则 1: 提供上下文而非控制

**错误做法:**
```
在 user.py 第 42 行添加一个 if 判断，检查 user.role == "admin"，
如果是就调用 grant_access()，否则调用 deny_access()
```

**正确做法:**
```
我们需要给 /dashboard 路由添加管理员权限检查。
背景: 目前所有登录用户都能访问 dashboard，但其中包含敏感的财务数据。
要求: 只有 admin 角色的用户可以访问，其他用户应该看到 403 页面。
参考: 现有的权限检查模式见 middleware/auth.py
```

区别在于: 第一种方式你在替 AI 思考实现细节；第二种方式你在解释"为什么"和"要达到什么效果"，让 AI 自己找到最佳实现方式。AI 可能找到比你预想的更好的方案。

### 原则 2: 信任但验证

给 AI 足够的自主权去完成任务，但在关键节点进行检查:

- **信任**: 不要在 AI 写每一行代码时都干预
- **验证**: 在任务完成后认真审查输出
- **反馈**: 发现问题时，说明问题和期望，让 AI 自己修正

```
# 好的验证节奏
1. 提交任务 -> 等待完成
2. 审查关键文件的 diff
3. 运行测试
4. 如有问题，描述问题让 AI 修复
5. 再次验证
```

不要:
```
# 每写一行都检查
"写完第一个函数了吗？给我看看"
"好，现在写第二个函数"
"等等，变量名改一下"
```

### 原则 3: 以成果为导向

关注最终结果，而非中间过程:

| 过程导向 (避免) | 成果导向 (推荐) |
|---|---|
| "用 for 循环遍历列表" | "过滤出有效的订单" |
| "创建一个叫 utils.py 的文件" | "我们需要一组字符串处理的工具函数" |
| "先写 class，再写 method" | "实现用户认证模块，要支持 JWT 和 session 两种方式" |

AI 可能用列表推导式而不是 for 循环，可能把函数放在 helpers.py 而不是 utils.py -- 这些都不重要，重要的是功能正确、代码质量好。

## 实践方法

### 1. 写好 CLAUDE.md (提供上下文)

CLAUDE.md 是你与 AI 之间最重要的沟通工具。它相当于你雇了一个新开发者时给他的入职文档。

**最小可用的 CLAUDE.md 结构:**

```markdown
# 项目名称

## 项目概述
一句话说明这个项目是什么、为谁服务。

## 技术栈
- 语言/框架/版本
- 数据库
- 部署环境

## 项目结构
关键目录和文件的说明 (不需要列出每一个文件)。

## 开发约定
- 代码风格 (formatter, linter 配置)
- 命名规范
- 测试要求
- Git 工作流

## 工作规则
1. 开始前读 PROGRESS.md
2. 完成后更新 PROGRESS.md
3. 每次有意义的变更都 commit
```

**关键细节别漏掉:**
- 环境变量的特殊处理 (比如 `env -u CLAUDECODE` 避免嵌套)
- API 代理地址
- 平台差异 (Mac vs Linux 的路径/命令差异)
- 不允许做的事情 (比如"不要修改 migration 文件")

### 2. 任务分解 (小而明确)

一个好的任务应该:
- **30 分钟以内能完成** (AI 时间，对应人类 2-4 小时的工作量)
- **有明确的完成标准** (怎么判断"做完了")
- **自包含** (不依赖其他正在进行的任务的输出)

**分解示例:**

原始需求: "给系统加上用户管理功能"

分解后:
```
任务 1: 创建 User 数据模型
  - SQLAlchemy model，字段: id, email, name, role, created_at
  - 包含基本的 CRUD 操作
  - 单元测试
  完成标准: pytest 全部通过

任务 2: 实现注册和登录 API
  - POST /auth/register
  - POST /auth/login (返回 JWT)
  - 密码使用 bcrypt 哈希
  完成标准: curl 测试注册和登录流程正常

任务 3: 实现用户列表和权限管理 API
  - GET /users (仅 admin)
  - PUT /users/{id}/role (仅 admin)
  - 使用 Task 2 的 JWT 做认证
  完成标准: 非 admin 用户调用返回 403

任务 4: 前端用户管理页面
  - 用户列表表格
  - 角色修改下拉框
  - 对接 Task 3 的 API
  完成标准: 页面可正常展示和修改用户角色
```

### 3. Review 而非微管理

有效的代码审查方式:

```
我检查了你的 PR，有几个问题:

1. auth.py 第 35 行: JWT 过期时间硬编码为 3600 秒，
   应该从配置读取，因为不同环境需要不同的过期时间。

2. user_model.py: 缺少 email 的唯一性约束，
   两个用户可以注册相同的邮箱。

3. 测试覆盖: 没有测试 token 过期的情况。

请修复这三个问题。其余部分看起来不错。
```

这种方式:
- 指出具体问题，而非泛泛而谈
- 解释为什么这是问题
- 不关注代码风格这类 linter 能自动处理的事情
- 肯定做得好的部分

### 4. 错误作为学习机会

当 AI 犯了错误，不要只是说"改正它"。写下来，让它不再犯:

```markdown
# 在 CLAUDE.md 中添加

## 已知陷阱
- 数据库迁移: 不要用 db.create_all()，必须用 alembic
  (因为生产环境已有数据，create_all 会丢失)
- API 响应: 所有时间字段使用 ISO 8601 格式 (UTC)
  (前端依赖这个格式解析)
- 文件上传: 最大 10MB，超过的要返回 413 而不是 500
  (之前因为没处理大文件导致 OOM)
```

把错误转化为 CLAUDE.md 中的规则，这样:
- AI 下次执行任务会自动遵守
- 新的 AI 会话也会知道这些陷阱
- 你不需要在每次任务中重复说明

## 反模式

### "事无巨细"型管理

```
# 反模式
"先创建文件 api.py"
(等完成)
"现在在里面写 import 语句"
(等完成)
"现在写路由函数的签名"
...
```

你在用最笨的方式使用最强的工具。让 AI 一口气完成整个模块。

### "甩手掌柜"型管理

```
# 反模式
"做完整个电商系统"
(三天后来看，发现完全不是想要的)
```

任务太大、缺少上下文、没有中间检查点。即使 AI 有能力做完，方向也可能偏离。

### "朝令夕改"型管理

```
# 反模式
"用 REST API"
(做到一半)
"改成 GraphQL 吧"
(又做到一半)
"还是 REST 吧，但是换个 URL 规范"
```

每次改变方向都浪费之前的工作。想清楚再提需求，或者在 Plan Mode 中先规划。

---

## 总结

管理 AI 代理的核心心法:

1. **你是上下文的提供者** -- 写好 CLAUDE.md，让 AI 知道它在什么环境中工作
2. **你是方向的把控者** -- 分解任务，定义完成标准，但不规定实现细节
3. **你是质量的守护者** -- 审查输出，提供反馈，把经验沉淀为规则
4. **你不是代码的编写者** -- 放手让 AI 发挥它的优势

记住: **Context, not control.**
