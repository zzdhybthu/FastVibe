# 常见反模式: 使用 Claude Code 时的陷阱与纠正

本文档列出使用 Claude Code (CC) 进行 agentic coding 时最常见的反模式。每个反模式包含描述、为什么有害、以及正确做法。

---

## 反模式 1: "做完这个项目"

### 描述

把一个巨大的、模糊的任务一股脑扔给 CC:

```
帮我做一个完整的电商系统，要有用户注册登录、商品管理、
购物车、订单、支付、物流追踪、评价系统、后台管理、数据报表。
```

### 为什么有害

- **上下文窗口溢出**: CC 的上下文窗口有限，超大任务会导致它遗忘前期的决策和代码
- **方向失控**: 没有中间检查点，CC 可能在错误的方向上走很远才被发现
- **质量下降**: 试图一次完成太多时，每个部分的质量都会降低
- **调试困难**: 最终输出如果有问题，很难定位是哪个环节出错
- **完全不可复现**: 如果中途失败，没有任何可用的中间产物

### 正确做法

将大任务分解为 **30 分钟以内可完成** 的小任务，每个有明确的完成标准:

```
任务: 创建 User 模型和注册 API
- User 模型: id, email, password_hash, name, created_at
- POST /api/auth/register: 接收 email + password，返回 user 对象
- 密码使用 bcrypt 哈希
- email 重复时返回 409
- 包含单元测试
完成标准: pytest 通过，curl 测试注册成功
```

经验法则: 如果你不能在一句话内说清楚任务的完成标准，任务就太大了。

## 反模式 2: 不提供上下文 (缺少 CLAUDE.md)

### 描述

不写 CLAUDE.md，不告诉 CC 项目的背景、技术栈、约定，直接开始派活:

```
给 API 加个缓存
```

### 为什么有害

- **猜测多于执行**: CC 不知道你用的是什么框架，会猜测或选择它认为最通用的方案
- **风格不统一**: 没有代码规范参考，CC 每次的输出风格可能不同
- **重复踩坑**: CC 不知道你之前踩过的坑，会再次犯同样的错误
- **效率低下**: 每次新会话，CC 都要重新理解项目，浪费大量上下文窗口

### 正确做法

在项目根目录创建 CLAUDE.md，至少包含:

```markdown
# 项目名称

## 概述
这是一个 [做什么的] 应用。

## 技术栈
Python 3.11, FastAPI, SQLAlchemy, PostgreSQL, Redis

## 关键文件
- src/api/ -- API 路由
- src/models/ -- 数据模型
- src/services/ -- 业务逻辑

## 开发规范
- 使用 black 格式化，isort 排序导入
- 所有 API 端点需要类型注解
- 缓存使用 Redis，键命名: {module}:{entity}:{id}

## 工作规则
1. 先读 PROGRESS.md
2. 做完更新 PROGRESS.md
3. 有意义的变更要 commit
```

投入 10 分钟写 CLAUDE.md，能在后续的每次任务中节省大量来回沟通。

## 反模式 3: 不检查输出 (盲信 AI)

### 描述

提交任务后，看到 CC 说"完成了"就直接用，不检查代码、不跑测试:

```
CC: "我已经实现了支付功能并通过了所有测试。"
人类: "好的" (直接部署)
```

### 为什么有害

- **安全隐患**: CC 可能引入 SQL 注入、硬编码密钥等安全问题而不自知
- **逻辑错误**: CC 可能误解了需求，实现了功能不等于实现了正确的功能
- **测试假象**: CC 写的测试可能只验证了 happy path，缺少边界条件和异常场景
- **技术债务**: 不审查的代码中可能有 hack、临时方案、TODO 变成了永久方案
- **生产事故**: 未经验证的代码进入生产环境，是事故的主要来源

### 正确做法

建立固定的验证流程:

```
1. 代码审查:
   - 看 git diff，理解 CC 实际改了什么
   - 重点关注: 安全相关代码、数据库操作、外部 API 调用

2. 运行测试:
   - 不只看 CC 写的新测试，也跑全量测试确保没有回归
   - 手动测试关键路径

3. 快速检查清单:
   - [ ] 没有硬编码的密钥/密码
   - [ ] 没有调试用的 print/console.log
   - [ ] 错误处理覆盖了关键路径
   - [ ] 没有绕过认证/授权的代码
   - [ ] 数据库查询使用了参数化
```

信任 AI 的能力，但验证 AI 的输出。这不是不信任，而是工程纪律。

## 反模式 4: 在同一目录运行多个 CC (文件冲突)

### 描述

在同一个项目目录里同时启动多个 CC 实例处理不同任务:

```
# 终端 1
cd /project && claude "实现用户模块"

# 终端 2
cd /project && claude "实现订单模块"
```

### 为什么有害

- **文件写入冲突**: 两个 CC 可能同时修改同一个文件 (比如 main.py 的路由注册、requirements.txt 的依赖)
- **Git 冲突**: 两个 CC 各自 commit，产生冲突的 git 历史
- **数据库锁**: 如果都在操作 SQLite，会遇到数据库锁定
- **不可预测的状态**: 一个 CC 读取文件时，另一个 CC 可能正在写入，导致读到半成品
- **调试噩梦**: 出问题后很难确定是哪个 CC 实例导致的

### 正确做法

使用 **Git Worktree** 隔离并行任务:

```bash
# 为每个任务创建独立的工作目录
git worktree add ../project-user-module feature/user-module
git worktree add ../project-order-module feature/order-module

# 终端 1: 在隔离的目录中工作
cd ../project-user-module && claude "实现用户模块"

# 终端 2: 在另一个隔离的目录中工作
cd ../project-order-module && claude "实现订单模块"

# 完成后合并
git merge feature/user-module
git merge feature/order-module
```

这样每个 CC 都有自己的文件系统视图，互不干扰。Stage 4 (worktree) 和 Stage 7 (orchestrator) 就是为此设计的。

## 反模式 5: 不使用 PROGRESS.md (丢失记忆)

### 描述

不维护 PROGRESS.md，每次新会话都从头开始:

```
# 第一天
"帮我建好数据库表和基本的 CRUD"
(CC 做完了，会话结束)

# 第二天，新会话
"继续做吧"
(CC: "继续做什么？我没有之前的上下文")

# 然后花 10 分钟解释昨天做了什么
```

### 为什么有害

- **上下文浪费**: 每次新会话都要花大量 token 重新解释现状
- **遗忘决策**: 之前做的架构决策、选型理由会丢失
- **重复工作**: CC 可能重新创建已经存在的文件或函数
- **不一致**: 没有状态记录，CC 可能做出与之前矛盾的决策
- **无法交接**: 如果切换到另一个 CC 会话，没有可传递的状态

### 正确做法

在 CLAUDE.md 中规定工作规则，并维护 PROGRESS.md:

```markdown
# PROGRESS.md

## 最后更新: 2024-03-15

## 项目阶段: MVP 开发 (第 2 周)

## 已完成
- [x] 数据库设计 (User, Order, Product 三张表)
- [x] User CRUD + JWT 认证
- [x] Product CRUD + 图片上传
- [x] 基本的前端框架搭建

## 进行中
- [ ] Order 模块
  - Order model 已创建 (src/models/order.py)
  - 状态机逻辑还没实现
  - API 端点还没创建

## 待办
- [ ] 购物车功能
- [ ] 支付集成 (决定用 Stripe)
- [ ] 部署配置

## 技术决策记录
- 2024-03-10: 选择 cursor-based 分页而非 offset-based，因为数据量可能很大
- 2024-03-12: 图片存储用本地文件系统，后续再迁移到 S3
- 2024-03-14: 订单状态流转用状态机模式，参考 src/services/state_machine.py

## 已知问题
- SQLite 在并发写入时偶尔报锁定错误，部署时需要换 PostgreSQL
- 前端的 CSS 在 Safari 下有布局问题 (低优先级)
```

每次任务完成后，让 CC 更新 PROGRESS.md。新会话开始时，CC 会自动读取这个文件恢复上下文。

## 反模式 6: 给出矛盾的指令

### 描述

在 CLAUDE.md、任务描述、或对话中给出互相矛盾的要求:

```markdown
# CLAUDE.md
代码风格: 使用 camelCase 命名

# 任务描述
按照 PEP 8 规范编写 Python 代码 (PEP 8 要求 snake_case)
```

或者:

```
"请用最简洁的代码实现"
(CC 写了精简的代码)
"为什么不写注释？代码要有详细的注释"
```

### 为什么有害

- **产出质量差**: CC 在矛盾的指令间左右为难，最终的输出是一个不伦不类的折中
- **来回修改**: 修正一个矛盾后触发另一个矛盾，陷入无限循环
- **信任破坏**: 反复否定 CC 的输出会让 CC 变得过于保守，不敢做任何决策
- **浪费 token**: 大量的上下文被用于处理矛盾，而非实际工作

### 正确做法

在 CLAUDE.md 中建立**权威的、无矛盾的规范**:

```markdown
## 代码规范 (权威来源，如有冲突以此为准)

### Python
- 命名: snake_case (PEP 8)
- 格式化: black
- 导入排序: isort
- 类型注解: 所有公开函数必须有

### 注释
- 函数: docstring (Google style)
- 行内注释: 只在逻辑不显然的地方添加
- TODO: 必须包含日期和原因
```

当你在任务描述中给出新指令时，检查是否与 CLAUDE.md 矛盾。如果是有意的覆盖，明确说明:

```
这个文件是例外情况，使用 camelCase 而不是 snake_case，
因为它是给 JavaScript 前端用的 JSON 序列化器。
```

## 反模式 7: 过度微管理 (每一步都干预)

### 描述

像用记事本一样一行一行地指挥 CC 写代码:

```
"创建 models.py 文件"
(等完成)
"导入 sqlalchemy"
(等完成)
"写一个 User 类继承 Base"
(等完成)
"加一个 id 字段，是 Integer 类型，主键"
(等完成)
"加一个 email 字段，String 类型，长度 255"
...
```

### 为什么有害

- **浪费 AI 能力**: CC 能在一次调用中写完整个模块，你在强迫它做文字打字机
- **极低效率**: 每次交互都有网络延迟和上下文重建的开销
- **打断思路**: CC 在宏观层面有更好的代码组织能力，逐行指令让它丧失全局视角
- **你的时间浪费**: 你花的时间比自己写代码还多
- **错失优化机会**: CC 可能有更好的实现方式，但你的逐行指令不给它发挥空间

### 正确做法

描述"要什么"，而不是"怎么做":

```
创建用户数据模型 (src/models/user.py):

字段:
- id: 自增主键
- email: 唯一索引，不可为空
- password_hash: 不可为空
- name: 可选
- role: 枚举 (admin/user)，默认 user
- is_active: 默认 True
- created_at: 自动设置
- updated_at: 自动更新

要求:
- 使用项目现有的 SQLAlchemy Base (src/db/base.py)
- 包含 to_dict() 方法 (排除 password_hash)
- 包含 verify_password() 和 set_password() 方法
- 单元测试
```

一次给完所有信息，让 CC 自己决定怎么组织代码。你可能会惊讶于 CC 写出比你预想更好的实现。

## 反模式 8: 忽视错误日志 (只说"不工作了")

### 描述

```
"API 不工作了，帮我修"
```

不提供任何错误信息、日志、复现步骤。

### 为什么有害

- **盲目调试**: CC 只能猜测问题在哪里，可能花大量时间检查无关的代码
- **制造新问题**: 没有准确的错误信息，CC 的"修复"可能修改了正常的代码
- **效率极低**: 一个 10 秒能定位的问题可能需要 10 分钟才能盲猜定位
- **多轮浪费**: "还是不行" "试试这个" "也不行" -- 不断来回直到碰巧对了

### 正确做法

提供完整的调试信息:

```
POST /api/orders 返回 500 Internal Server Error。

请求:
curl -X POST http://localhost:8000/api/orders \
  -H "Authorization: Bearer xxx" \
  -H "Content-Type: application/json" \
  -d '{"product_id": 1, "quantity": 2}'

错误日志 (来自 uvicorn 终端):
```
[ERROR] 2024-03-15 10:30:00 - Traceback (most recent call last):
  File "src/api/orders.py", line 45, in create_order
    order = OrderService.create(user_id, data)
  File "src/services/order_service.py", line 23, in create
    product = db.query(Product).get(product_id)
AttributeError: 'Query' object has no attribute 'get'
```

环境: Python 3.11, SQLAlchemy 2.0, FastAPI 0.104
注意: 这个项目最近从 SQLAlchemy 1.4 升级到 2.0
```

有了这些信息，CC 能立刻看出是 SQLAlchemy 2.0 API 变更导致的，直接给出准确修复。

## 反模式总结速查表

| 反模式 | 一句话识别 | 正确做法 |
|---|---|---|
| 任务太大 | "做完这个项目" | 分解为 30 分钟可完成的小任务 |
| 缺少上下文 | 没有 CLAUDE.md | 10 分钟写好 CLAUDE.md |
| 盲信输出 | 不跑测试就部署 | 审查 diff + 跑测试 + 手动验证 |
| 目录冲突 | 同目录多个 CC | 用 git worktree 隔离 |
| 丢失记忆 | 每次重新解释 | 维护 PROGRESS.md |
| 矛盾指令 | CLAUDE.md 和任务说的不一样 | 统一规范，明确覆盖 |
| 过度微管理 | 一行一行指挥 | 描述"要什么"而非"怎么做" |
| 忽视日志 | "不工作了" | 提供完整的错误日志和复现步骤 |

## 建立正确的心智模型

避免所有反模式的关键在于建立正确的心智模型:

**CC 是一个极其有能力但没有项目上下文的高级工程师。**

- 它不需要你告诉它怎么写代码 (不要微管理)
- 它需要你告诉它项目的背景和约定 (提供 CLAUDE.md)
- 它能处理复杂任务但需要明确的范围 (分解任务)
- 它的输出需要验证但通常质量不错 (信任但验证)
- 它没有记忆但能快速恢复 (维护 PROGRESS.md)
- 它会严格执行你的指令，所以指令不能矛盾 (统一规范)
- 它不能读你的脑子，出了问题要给足够信息 (提供日志)
- 它需要安全的工作环境 (隔离并行实例)

把 CC 当作团队成员而非工具，你会自然地避免大多数反模式。
